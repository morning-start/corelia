# 系统架构

> Corelia 采用清晰的前后端分离架构，通过 Tauri 实现桌面端能力

## 总览

Corelia 是一款基于 Tauri + SvelteKit + Rust 的桌面快速启动器，采用三层架构：

```
┌─────────────────────────────────────────────┐
│         WebView 前端 (SvelteKit)            │
│  UI展示 · 状态管理 · 搜索逻辑 · WASM运行    │
└──────────────────┬──────────────────────────┘
                   │ Tauri IPC Bridge
┌──────────────────┴──────────────────────────┐
│         Rust 后端                             │
│  插件系统 · 系统服务 · 能力暴露              │
└─────────────────────────────────────────────┘
```

## 技术选型

| 层级 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 前端框架 | SvelteKit | 2.x | 前端应用框架 |
| 前端语言 | TypeScript | ~5.6.2 | 类型安全 |
| 构建工具 | Vite | 6.x | 前端构建 |
| 桌面框架 | Tauri | 2.x | 跨平台桌面壳 |
| 后端语言 | Rust | 1.94.0 | 高性能核心逻辑 |
| JS 引擎 | rquickjs | 0.11.0 | 插件沙箱运行 |
| 包管理 | Bun | 1.3+ | 依赖管理和运行时 |

## 架构分层

### 前端层 (WebView)

**职责**：用户界面、交互处理、状态管理、搜索逻辑、WASM执行

**核心模块**：
- **UI组件** - 搜索框、结果列表、设置面板等用户界面
- **状态管理** - 应用全局状态、搜索状态、主题配置（Store 层，仅负责响应式状态）
- **搜索系统** - 模糊匹配、结果合并、排序逻辑
- **插件前端层** - WASM Patch加载、插件服务集成（Service 层，负责业务逻辑）
- **错误展示** - 用户友好的错误提示，不记录日志

**前端不应该做的**：
- ❌ 维护插件 VM 状态（统一由后端管理）
- ❌ 持久化数据（交由后端存储）
- ❌ 直接操作文件系统（通过后端 API）

### 后端层 (Rust)

**职责**：系统能力提供、插件管理、API暴露、资源管理

**核心模块**：
- **Commands 层** - Tauri命令暴露层（仅做参数验证和转发）
- **Services 层** - 业务逻辑层（各 Service 单一职责，独立可测试）
- **插件系统** - 插件扫描、加载、生命周期管理
- **QuickJS运行时** - JS沙箱、VM池化、代码执行
- **ApiBridge** - utools兼容API注入（仅负责API注入，不包含业务逻辑）
- **WasmBridge** - QuickJS与WebView之间WASM调用桥接
- **系统服务** - 配置管理、剪贴板、自启动、窗口控制等
- **错误处理** - 记录和上报错误，不直接展示给用户

**后端不应该做的**：
- ❌ 包含 UI 展示逻辑
- ❌ 直接操作 WebView 内容（通过事件通信）

## 前后端职责边界

### 清晰的职责划分

```
┌─────────────────────────────────────────────────────────────┐
│                        前端层                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ UI 渲染 & 交互                                           │
│  ✅ 用户输入处理 (键盘、鼠标)                                │
│  ✅ 搜索结果的前端排序 & 展示                               │
│  ✅ 响应式状态管理 (Store 层)                                │
│  ✅ 错误友好展示 (Toast、文字提示)                          │
│  ✅ WASM 代码执行 (在 WebView 中)                            │
│                                                             │
│  ❌ 插件 VM 管理（由后端统一管理）                          │
│  ❌ 数据持久化（调用后端 API）                               │
│  ❌ 直接文件系统操作（通过后端 API）                        │
│                                                             │
└─────────────────────────────┬───────────────────────────────┘
                              │ Tauri IPC
┌─────────────────────────────┴───────────────────────────────┐
│                        后端层                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 系统能力封装 (剪贴板、窗口、文件系统)                    │
│  ✅ 插件加载、VM 管理、生命周期管理                          │
│  ✅ 配置持久化 & 读取                                       │
│  ✅ 错误记录 & 上报                                          │
│  ✅ 资源池管理 (VM 池、连接池等)                            │
│                                                             │
│  ❌ 直接 UI 操作（通过事件通知前端）                        │
│  ❌ 用户交互逻辑（留给前端）                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 数据流示例

#### 搜索流程
```
用户输入 ───────────────────┐
                           ▼
[前端] 输入处理 ────────────┤
                           ▼
[前端] Store 更新 (响应式) ──┐
                           ▼
[前端] Service 层触发搜索 ─┬───┐
                          │   │
                 [系统搜索]  [插件搜索]
                    │          │
                 [本地]   [并行调用后端]
                    │          │
                    └────┬─────┘
                         ▼
                    [结果合并]
                         ▼
                    [前端展示]
```

#### 配置管理流程
```
用户修改设置 ──────────────┐
                           ▼
[前端] 更新 Store 状态 ─────┤
                           ▼
[前端] 调用后端命令 ────────┤
                           ▼
[后端 Commands] 参数验证 ──┤
                           ▼
[后端 Service] 业务逻辑 ───┤
                           ▼
[后端] 持久化到磁盘 ────────┘
                           (状态同步完成)
```

#### 插件执行流程
```
用户选择插件结果 ─────────┐
                           ▼
[前端] 调用 plugin_execute ─┐
                           ▼
[后端 Commands] 验证参数 ──┤
                           ▼
[后端 PluginService] ──────┤
                           ├── 检查插件状态
                           ├── 获取插件 VM
                           ├── 验证状态 (Ready/Cached)
                           └── 执行代码
                           ▼
[后端 QuickJS Runtime] ────┤
                           ├── 查找 VM
                           ├── 执行 JS 代码
                           └── 返回结果
                           ▼
[后端 Service] 返回结果 ───┐
                           ▼
[前端] 展示结果/错误 ───────┘
```

## 错误处理策略

### 统一错误处理原则

**前端负责**：
- ✅ 友好的错误展示（Toast、文字提示）
- ✅ 错误边界处理（避免整个应用崩溃）
- ✅ 错误重试逻辑（按需）
- ❌ 不记录错误日志（通过事件上报给后端）

**后端负责**：
- ✅ 错误记录（标准输出/日志文件）
- ✅ 错误上报（可选：Sentry等服务）
- ✅ 错误分类和包装
- ❌ 不直接展示错误给用户（返回结构化错误信息）

### 错误数据结构

后端返回统一格式：
```typescript
{
  success: boolean;
  data?: T;
  error?: {
    code: string;       // 错误码
    message: string;    // 用户友好的消息
    details?: string;   // 技术细节（开发模式）
  }
}
```

## 配置管理职责

### 三层配置模型

| 配置层级 | 存储位置 | 职责 | 可修改？ |
|---------|---------|------|---------|
| **System** | `config.system.json` | 系统级设置 (快捷键、启动项等) | 用户可修改 |
| **User** | `config.user.json` | 用户个性化设置 (主题、窗口大小等) | 用户可修改 |
| **App** | `cache/app.json` | 应用运行时数据 (历史、插件缓存等) | 自动管理 |

### 职责划分

**后端负责**：
- ✅ 配置持久化到磁盘
- ✅ 配置读取
- ✅ 配置验证
- ✅ 默认配置提供
- ✅ 配置重置功能

**前端负责**：
- ✅ 配置展示 (UI组件)
- ✅ 配置编辑 (表单、交互)
- ✅ 配置预览 (即时反馈)
- ✅ 调用后端 API 保存/加载
- ✅ 本地缓存配置状态 (Store 层)

## 数据流

### 搜索流程

```
用户输入 → 前端搜索处理 → 多源结果合并 → UI渲染
                 ↓
         调用后端API（如需要）
```

### 插件加载流程

```
前端触发加载 → 后端创建VM → 注入API → 执行插件代码 → 插件就绪
                          ↓
                  通知前端加载WASM Patch
```

### WASM调用流程

```
插件代码调用 → 后端WasmBridge → 前端执行WASM → 结果返回后端 → 插件获取结果
```

## 关键设计决策

1. **前后端职责分离** - UI和状态管理在前端，系统能力和插件管理在后端
2. **QuickJS而非V8** - 更轻量，适合沙箱化插件执行
3. **VM池化** - 插件独立VM，支持闲置回收，控制资源上限
4. **WASM IPC桥接** - QuickJS无法直接运行WASM，通过Tauri事件转发到WebView
5. **插件懒加载** - 仅扫描元数据，按需加载执行
6. **透明窗口设计** - 无边框、失焦自动隐藏
